# 管理后台产品功能规划书（可迭代路线图）

## 1. 产品目标
降低运营维护成本：卡密、调优、反馈统计能够“少操作、少出错、可追溯”
提升可控性：调优迭代要能回滚、要能灰度/对照
提升可视化：准确率与失败原因要能定位到“策略/版本/场景/数据段”
强化安全性：鉴权、审计、最小权限、敏感操作保护
## 2. 典型用户
运营同学：批量生成/废除卡密、查看反馈统计、导出报表
算法/调优同学：查看调优历史、对 device/技能参数做迭代、回滚
管理员：权限管理、审计、配置开关、紧急处置（泄漏、异常行为）
##3. 模块规划（按成熟度拆）
A. Gift codes（卡密运营）
MVP 已有：生成、列表展示、废除（如果你后端提供）、分页

迭代建议：

过滤与导出：按状态/时间/前缀/使用设备筛选，导出 CSV
批量废除：支持“选中若干 + 二次确认 + 导出影响范围”
生成任务管理：生成任务队列状态（进行中/完成/跳过原因），避免只给明文结果
“最小可用提示”：泄漏风险提示、过期/未使用占比提示（运营友好）
B. Feedback（反馈统计）
MVP 已有：按天回看、准确率总览、最近反馈时间线

迭代建议：

详情标注面板：对单条反馈支持“备注/归因标签/复核 verdict”
分维度分析：按渠道/场景/关系阶段/透镜证据密度区间分布（便于知道哪里错）
数据闭环：把“错误样本”回流到调优（比如：哪些聊天模式导致误判）
C. Tuning（调优配置）
MVP 已有：写入调优、重置、查看调优明细（分页后更清晰）

迭代建议（关键）：

版本化调优：每次写入生成一个 tuning_version，支持回滚到任意版本
灰度发布：将 device 分组，控制新参数比例（A/B 或 canary）
影响评估：在发布前后对准确率/失败率做对照图表
安全校验：输入校验更严格（参数上限、必填规则、异常提示）
D. Auth & Security（鉴权与安全）
MVP：token 存 sessionStorage + 简单守卫

迭代建议：

角色权限：运营/调优/管理员不同权限
审计日志：记录关键操作（生成/废除/调优/重置/导出/登录）
Token 轮换与过期策略：避免长期有效 token 风险
扫码登录/二维码：登录页已经拆出来，下一步接扫码获取 token -> 跳转即可
E. Admin UX（运营体验）
迭代建议：

统一布局规范：左侧导航 + 顶栏信息 + 内容区最大宽度 + 统一卡片间距
关键按钮保护：所有“高风险动作”都二次确认 + 明确影响说明
空状态与错误状态：接口失败时给清晰文案与重试入口
运行中状态：长请求有 loading 与“不会卡死”的提示
4. 里程碑规划（建议节奏）
里程碑 1（近期）：完善筛选/导出、反馈复核标签、调优版本号（最小版本化）
里程碑 2（中期）：调优灰度发布 + 回滚 + 发布对照图
里程碑 3（后期）：审计日志 + RBAC 权限 + 扫码登录完整闭环
5. 验收标准（可量化）
运营操作平均耗时下降（例如生成/废除/导出不超过 N 步）
调优回滚成功率提升（出现异常能在 T 分钟内回滚）
反馈正确率提升或误判率下降，并能在分维度中观察到改善
安全事件响应流程可执行（审计可查 + 权限可控）
